Introduction

Apache Spark is a unified analytics engine for large-scale data processing. One of its key components, Spark Structured Streaming, is a scalable and fault-tolerant stream processing engine built on the Spark SQL engine. It allows you to process real-time data streams using high-level declarative APIs, which are integrated with the Spark ecosystem. This makes it possible to perform complex streaming computations, including joining streams with batch data and running arbitrary computations using the DataFrame and Dataset APIs.

In this tutorial, we will explore Spark Structured Streaming by demonstrating how to use different data sources. We will cover the rate data source, file streaming data source, and Kafka source, providing Scala code examples for each.

What is Spark Structured Streaming?

Apache Spark Streaming – an extension of the core Spark API – is a scalable and fault-tolerant system that natively supports batch and streaming workloads.

Data is ingested from sources such as file, Amazon Kinesis, Kafka or TCP sockets, and processed using complex algorithms. The processed data is then pushed out to file systems, databases, and live dashboards. Since Spark Streaming was introduced, an easier-to-use streaming engine called Apache Spark Structured Streaming has been introduced for streaming applications and pipelines.

Apache Spark Structured Streaming allows you to take batch mode operations that are conducted using Spark’s structured APIs, and run them in a streaming fashion. What are the benefits? Reduced latency, incremental processing, and rapid value with virtually no code changes.

Spark Structured Streaming uses the Dataframe or Dataset APIs. Dataframe is a generic row type and has a higher level of abstraction from the RDDs. Datasets are the same as the Dataframe but provide type safety. Dataset is a declarative language that operates on domain objects and offers complex operations on data with a defined schema.

Introduced starting with Spark 2.x, it lets you apply all the SQL-like operations, (unlike Spark Streaming which is RDDs based). Also, it’s closer to real-time streaming as each row is processed and the outcome is appended to an unbounded results table.

By default, the queries in Structured Streaming are processed using micro-batch processing engine. Here, data streams are processed as a series of small batch jobs, thereby achieving end-to-end latencies as low as 100 milliseconds and exactly-once fault-tolerance guarantees.

A new low-latency processing mode called Continuous Processing was introduced since Spark 2.3. This processing mode can achieve end-to-end latencies as low as 1 milliseconds with at-least-once guarantees.

Input Sources

Spark Streaming ingests data from different types of input sources for processing in real-time.

  • Rate (for Testing): It will automatically generate data including 2 columns timestamp and value . This is generally used for testing purposes.
  • Socket (for Testing): This data source will listen to the specified socket and ingest any data into Spark Streaming. It is also used only for testing purposes.
  • File: This will listen to a particular directory as streaming data. It supports file formats like CSV, JSON, ORC, and Parquet. You can find the latest supported file format list here.
  • Kafka: This will read data from Apache Kafka® and is compatible with Kafka broker versions 0.10.0 or higher.

Output Sinks

In Spark Streaming, output sinks store results into external storage.

  • Console sink: Displays the content of the DataFrame to console. In this tutorial, we have only used console sink.
  • File sink: Stores the contents of a DataFrame in a file within a directory. Supported file formats are csv, json, orc, and parquet.
  • Kafka sink: Publishes data to a Kafka® topic.
  • Foreach sink: Applies to each row of a DataFrame and can be used when writing custom logic to store data.
  • ForeachBatch sink: Applies to each micro-batch of a DataFrame and also can be used when writing custom logic to store data.

Output modes

After processing the streaming data, Spark needs to store it somewhere on persistent storage. Spark uses various output modes to store the streaming data.

  • Append Mode: In this mode, Spark will output only newly processed rows since the last trigger.
  • Update Mode: In this mode, Spark will output only updated rows since the last trigger. If we are not using aggregation on streaming data (meaning previous records can’t be updated) then it will behave similarly to append mode.
  • Complete Mode: In this mode, Spark will output all the rows it has processed so far.

Rate Data Source

The rate data source is a built-in data source in Spark Structured Streaming, which generates data at a specified rate. This is useful for testing and benchmarking streaming applications.

Using the Rate Data Source

In the following example, we create a streaming DataFrame from the rate data source, generating 10 rows per second. Each row contains a timestamp and an incrementing value. We then transform the stream by adding a column ‘result‘ by just adding 1 to column value and write the transformed stream to the console.

File Streaming Data Source

With file input source, our application will wait for available data in the specified directory. We will use some of the stock data available on your sandbox. Spark reads files written in this directory as a stream of data. Files will be processed in the order of file modification time. If the option latestFirst is set to true, the order will be reversed. Supported file formats are text, CSV, JSON, ORC, Parquet.

Using the File Streaming Data Source

Our data contains the fields Date,Open,High,Low,Close,Adj Close,Volume and we will extract Name from the filename and add it as a new column to the stream Dataframe.

Kafka Source

Apache Kafka is a distributed streaming platform that is commonly used for building real-time data pipelines and streaming applications. Spark Structured Streaming integrates with Kafka, allowing you to read data from and write data to Kafka topics.

Using the Kafka Data Source

To use Structured Streaming with Kafka, developers can leverage the Kafka source that is built into Spark, which provides a fault-tolerant and scalable way to read data from Kafka topics. The Kafka source is integrated with Spark’s Structured Streaming API, allowing developers to easily define streaming queries on Kafka data using the DataFrame API.

To read data from Kafka, developers can use the `readStream` method and specify the Kafka source along with the Kafka broker details and the topic to read data from. Once the data is read into a DataFrame, developers can apply transformations and aggregations on the data using Spark’s DataFrame API.

In this example, we will build a real-time data pipeline that reads data from a Kafka topic, processes it using Spark Structured Streaming, and writes the processed data to a console sink. This example will illustrate the end-to-end process of creating a streaming application with Spark.

Step-by-Step Guide:

  • Set Up Kafka Topic: First, ensure that your Kafka server is running and create a Kafka topic named sensor-data to simulate streaming data from IoT sensors.
  • Produce Data to Kafka Topic: Use a Kafka producer to send simulated sensor data to the sensor-data topic. You can use the Kafka console producer for this purpose.
  • Build the Spark Structured Streaming Application: Create a Scala application to read from the Kafka topic, process the data, and write the results to the console.

Summary

Spark Structured Streaming provides a robust framework for processing real-time data streams using high-level APIs. In this tutorial, we covered three different data sources: the rate data source, file streaming data source, and Kafka source. Each example demonstrated how to create a streaming DataFrame, transform the stream, and write the output to the console using Scala code. Understanding these concepts and examples will help you build and deploy real-time streaming applications using Spark Structured Streaming.
